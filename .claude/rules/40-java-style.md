# Java 风格与测试命名规则

## 总原则

编码风格默认遵循 Google Java Style，并优先保证：

- 边界清晰
- 命名可读
- 文件聚焦
- 错误处理明确
- 对较弱模型友好

## 生产代码

- 生产代码标识符、包名与公开 API 默认使用英文
- 中文用于仓库文档、流程资产、评审说明与验证结果
- 不得为了“看起来整洁”破坏层边界
- 不得通过复制粘贴跨层映射逻辑规避 mapper 设计
- 复杂业务方法默认采用步骤性注释（如 `// 1.`、`// 2.`、`// 3.`）明确阶段意图，便于代码审查与企业团队协作
- 但步骤性注释必须服务于真实业务编排；简单直译方法、getter/setter、纯胶水代码不得滥用步骤编号

## 测试命名

JUnit 5 测试方法默认采用 BDD 风格：

- `should_succeed_when_*`
- `should_fail_when_*`

如确有历史兼容原因，可接受等价的 `should_*_when_*` 变体，但必须语义清晰。

## 测试文档化

每个非平凡测试默认应具备：

- `@DisplayName`
- 紧邻测试的 JavaDoc

测试 JavaDoc 应说明：

1. 场景
2. 预期
3. 核心断言/观察点

注意：验证结果本身属于运行证据，应记录在验证产物，而不是伪装成源码中的“实际结果”。

## 测试分层

- `*Test`：单元测试
- `*IT`：小型集成测试
- `*E2E`：后端 API E2E

本项目当前不做前端点击测试；E2E 指后端 API 场景编排。

## 文件与函数

- 优先小文件、单一职责
- 超大文件应视为设计信号，而不是默认常态
- 复杂编排方法可以写步骤性注释；简单 getter/setter/直译方法不要滥用注释

## 例外

任何偏离本规则的长期例外，必须：

- 记录原因
- 指明作用范围
- 在 design 或 waiver 中可追溯
