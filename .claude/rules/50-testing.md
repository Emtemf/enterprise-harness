# 测试与验证规则

## TDD 基线

本项目默认采用 TDD：

1. 先写失败测试
2. 亲眼看到失败
3. 写最小实现
4. 看到测试转绿
5. 仅在全绿时重构

没有观察到有效 RED，不得声称完成了 TDD。

## 最小测试类型

对 Java 后端，至少区分：

- 单元测试：核心 application / domain 逻辑
- 集成测试：持久化、小范围控制器路径
- API E2E：真实 HTTP 场景编排，不涉及 UI 点击

## 覆盖率目标

Phase 2 起，单元测试 line coverage 目标为 **85% 以上**。

覆盖率门禁应由 JaCoCo check 等机械手段执行；不得用测试数量替代覆盖率。

## API E2E 定义

后端 API E2E 的最低标准：

- 启动真实 HTTP 端口
- 发真实请求
- 断言 response
- 断言必要的持久化最终状态

Swagger UI 可访问不等于 E2E 通过。

## 验证证据

任何“已完成 / 已修复 / 已通过”的声明，都必须由新鲜验证证据支撑。

最低证据应包括：

- 运行了什么命令
- 命令输出或关键结果
- 失败/跳过项是什么

## 禁止事项

- 不得跳过 RED 直接改生产代码
- 不得只跑局部检查却声称全部通过
- 不得把旧测试报告当成当前完成证据
- 不得用 MockMvc 冒充真实 HTTP E2E
